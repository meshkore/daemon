- 1.30.4 — TEAM-EXT CAP + STALE-REQUEST SELF-HEAL, PORT-STABILITY DOC FIX (operator field report 2026-07-08, one incident, two fixes). A shared-daemon restart moved the daemon from port 5573 (where it had been running for the whole session, and where the Reddit project's `consultant` client hardcoded its expectation) to 5570 per the documented playbook, breaking that external integration with a 502. Root-caused, daemon restarted back on 5573 with the operator's confirmation. Two follow-ups from the incident: **(a) `teamext.py` self-heal.** A `_teamext_watch` observer thread has no persisted registry of its own — when the daemon PROCESS dies mid-turn (the restart above, or any crash/`/shutdown`), the thread dies with it and its request sits `queued`/`running` in `.meshkore/.runtime/team-requests.json` forever, silently occupying an `EXT_ASK_CONCURRENT_CAP` slot until the 24h GC eventually erases the entry outright (found live: one such orphan, ~2h old, was enough to trip the cap). New `_teamext_reap_stale()` — age is the tell, since no legitimate watcher survives past its own `_WATCH_MAX_SECS` (1h) hard-timeout branch — flips any `queued`/`running` entry older than that to `error`, called at both read sites (`POST /team/<id>/ask`'s cap check, `GET /team/requests/<rid>`'s poll) so a client polling an orphan gets a terminal status immediately instead of hanging, and the cap slot frees right away instead of after 24h. Also raised `EXT_ASK_CONCURRENT_CAP` 2 → 6 (operator: a single local operator legitimately fires several ask/poll probes back-to-back while debugging; 2 was tight enough that ONE orphan alone locked the member out). New `tests/test_teamext_reap.py` (pure-logic, no daemon boot): stale running→error, fresh running left alone, terminal states never touched, cap raised. **(b) `scaffold.py` doc fix.** The generated `cluster.yaml`'s transport comment claimed "it may shift between restarts — that is normal" — false since py-1.15.0's sticky-port registry (`daemon-port-stability`/D-PORT-01) shipped, but nobody had corrected the template, which is exactly why a real drift wasn't suspected as a bug sooner. Comment now describes the actual sticky behavior; new templated `architect.port` block seeds `_pick_port`'s from-scratch fallback (only consulted when the registry has no entry yet for this cluster_id) so a wiped registry or a first boot on a new machine lands on a durable, intentional port instead of "first free in range". `meshkore-server/.meshkore/public/cluster.yaml` (not in the bundle — local, ungitted) hand-pinned to `architect.port: 5573` to match. Neither fix touches the wire protocol or `_pick_port`'s core algorithm. Suite 297 passed.
- 1.30.3 — SERVER-HOME NEVER A PROJECT (durable, env-independent; operator field report 2026-07-07: "solved 20 times, keeps coming back"). The central daemon's own home (`meshkore-server` — its `.meshkore` IS the machine-global store: ideas, the projects registry, external creds) kept reappearing in `/projects` and the cockpit rail as if it were a project. **Root cause:** the only exclusion signal was the STRUCTURAL test `_is_home_context` (compares the resolved context root's `.meshkore` to the global-ledger root) — which is only correct when `MESHKORE_GLOBAL_ROOT` resolves to the home's `.meshkore`. Launch the daemon any other way and the test returns False, so the home is treated as a real project. That launch-dependence is exactly why it returned after differently-launched restarts. **Fix — an env-independent id backstop layered on the structural test:** `projectsapi._DEFAULT_HOME_IDS = {"meshkore-server"}` (extendable per machine via `MESHKORE_HOME_IDS`), combined in one `is_home(pid, root)` gate used on EVERY /projects path: `_real_project_ids`/`projects_list` exclude it; `rehydrate_projects` never re-registers a home id from a stale `projects.json` and calls the new `_prune_home_from_projects` self-heal (rewrites the file without any home row, no-op on a clean/empty file); `project_register` returns 409 (the home can't be adopted). `readapi` `/health.server_home` now ORs the id denylist too, so the cockpit's `markHome` gets a reliable signal regardless of launch flags. Cockpit companion (`known-projects.ts`, NOT in the bundle): the same id is hardcoded into the client-side home set so the rail filters it even if a `/health` signal is ever missed — unconditional on both ends. Pure exclusion/guard logic — no wire/schema change. `tests/test_multiproject.test_server_home_id_backstop` proves exclusion + 409 when the structural test is deliberately blind.
